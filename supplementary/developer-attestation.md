## Commit Authorship

Since discovering that Claude Code could generate git commits, I have made very few — possibly zero or near-zero — manual commits. Nearly all commits across all repositories analyzed in this study were generated by Claude Code (initially ungoverned, then governed via the Triad protocol).

## Work Method

Except for external correspondence, some Traveller RPG game content, and portions of the book manuscript, I typed almost nothing directly into code or content files. Nearly all development work was conducted through prompts to Claude Code (initially unnamed, later designated "Argus"), giving instructions. The Triad protocol (Auditor/Generator role separation) was used for most substantive work once established.

## Triad Usage by Project

**Book (relinquishment repo):** I used the Generator role exclusively for generating manuscript content. On occasion I used the Auditor role directly for small, targeted, specific changes — I acknowledge this as bypassing the protocol ("cheating"). All Generator prompts are preserved in the plans/ directory of the relinquishment repository, making Triad usage independently verifiable from the commit history and plan files.

**Traveller (traveller-VTT-private repo):** I often skipped the Generator role, working directly without role separation. Exception: for larger tasks I would sometimes use the full Triad with Generator prompts. Triad usage in this repo was inconsistent and task-size-dependent.

**Governance system (aurasys-memory repo):** Mixed. Memory maintenance and protocol updates were often done without formal Triad. Substantive changes (Plan 0308, Plan 0299, etc.) used full Triad.

## Implications for the Study

1. **Co-author tag reliability:** The `Co-Authored-By: Claude` tags in commit messages are reliable indicators of AI-generated commits. Manual commits without this tag are rare.
2. **Commit pattern validity:** Structural breaks detected in commit metrics (message length, lines changed, file count) reflect changes in the AI system's behavior under different governance regimes, not changes in my typing patterns.
3. **Governance identification:** The presence or absence of Triad discipline should be identifiable from commit data patterns (structured plan-phase commits vs. unstructured bulk commits), independent of explicit governance metadata.
4. **Governance gradient:** The projects form a natural gradient of Triad usage intensity: Baseline (zero) → Traveller (inconsistent, task-dependent) → Governance system (mixed) → Book (consistent, verifiable from preserved prompts). This gradient is a feature of the natural workflow, not an experimental design.
5. **Verifiability:** The book project's Generator prompts are preserved as plan files, providing independent verification of Triad discipline claims.
